Evaluating token reuse feasibility (16 seconds)
Good question — let me answer this precisely as a senior dev would.

The short answer
Yes, you can reuse the token — but only for testing. Here's the full picture:

How Meta tokens actually work
A Meta access token is tied to a WhatsApp Business Account (WABA), not to a single phone number. One token can send/receive messages across multiple phone numbers, as long as all those numbers are registered under the same WABA.

Your current setup:

Bot	Phone Number ID	Phone Number
BTBIZ Leadgen	928192920367933	+91 90000 01870
Healthcare	902515902955901	+91 9625887227
Pharmacy (new)	New phone_number_id	New number needed
The token is shared — the routing is done by phone_number_id in the webhook payload. Our codebase already does exactly this in server/whatsapp.ts.

What you need to reuse it
A new WhatsApp-registered phone number — you cannot reuse +91 9625887227 or +91 90000 01870. Each bot needs its own number.
Add that number to the same WABA — in Meta Business Manager, add a new phone number to the existing WhatsApp Business Account.
Register the same webhook URL — https://btbizaichatbot.replit.app/webhook already receives all messages from all numbers in the WABA.
The same token will work automatically for the new number once it's added to the WABA.

Where does the pharmacy data go?
If you want it in THIS codebase (easiest for testing):

Add a new row to the clients table (client_id=7, phone_number_id = new pharmacy number)
Add a new routing branch in server/whatsapp.ts — same pattern as the Healthcare bot
Create a pharmacy_conversations table for state
If your pharmacy system has its own backend (correct for production):

The webhook still hits this server
This server just forwards the message to your pharmacy backend API
Your pharmacy backend processes it and returns a reply
This server sends the reply via Meta API using the shared token
Positives ✅
Why it helps
One Meta App approval	No need to go through Meta's business verification again
Same webhook endpoint	No new server setup, no new SSL, no new domain
Multi-tenant already built	server/whatsapp.ts routes by phone_number_id — just add a new case
Faster testing	Token is live, approved, and working today
Lower cost	One Meta subscription covers all numbers
Negatives / Risks ⚠️
Risk	Impact	Mitigation
Token expires → all 3 bots go down	High	Use permanent System User token (already done for Healthcare) — never temporary
Shared WABA messaging rate limits	Medium	Meta limits messages per WABA per day — all 3 bots share that quota
One Meta App review rejection affects all	Low	Already approved — don't change app settings
Data mixing risk	Medium	Strictly separate by client_id and separate conversation tables — never query across bots
Pharmacy outage could affect other bots	Low if isolated	Keep pharmacy handler in its own file, wrap in try/catch
My recommendation (senior dev view)
For testing right now:
Use the existing token. Add the pharmacy phone number to the same WABA. Route it in server/whatsapp.ts. Store data in a new pharmacy_conversations table. Fast, zero cost, working in hours.

For production later:
Get a separate WABA and token for the pharmacy. Reasons: the pharmacy may be a different legal entity or client, you may want to sell it independently, and one security incident shouldn't take down all three bots.

One thing to check first
You need a new phone number that isn't already registered on WhatsApp personally. Can be a SIM, a VoIP number, or a landline — Meta accepts all three. Do you have a number ready, or do you need guidance on getting one?

Planning system testing strategy (8 seconds)
Straight answer — let me address both options you suggested:

Option 1 — if (phoneNumberId === LEAD_NUMBER_ID)
This won't work for your scenario. You're proposing to share the Healthcare bot's number for pharmacy testing. That means both bots have the same phone_number_id. The if-check can't distinguish them — both messages arrive with identical phone_number_id. You'd have no flag to split on.

This approach does work — but only when you have two different phone numbers. That's exactly how the existing two bots are separated today in server/whatsapp.ts:

if (phoneNumberId === HEALTHCARE_PHONE_NUMBER_ID) {
  return processHealthcareMessage(...)
}
// else → BTBIZ leadgen

Option 2 — TEST_MODE = true
This breaks your live Healthcare bot entirely. Every real patient who messages the Healthcare number (+91 9625887227) would be routed to your pharmacy code instead. Real bookings would fail. You'd be running a live experiment on real patients.

The real problem with sharing the Healthcare number
You can't tell a "pharmacy test message" apart from a "real patient message." They look identical to the server. There's no magic flag inside the WhatsApp message itself.

Patient sends: "hello"
Pharmacy tester sends: "hello"
→ Server receives both.
→ Identical payload.
→ No way to know which is which.

What actually solves this cleanly
Get a second phone number — any number. It does not need to be a new SIM or cost money. Options:

Option	Cost	Time to set up
Your personal number (temporarily)	Free	5 minutes
Any spare SIM	Free	5 minutes
Google Voice / TextNow (VoIP)	Free	10 minutes
JustCall / Twilio number	~$1/month	15 minutes
Add it to the same WABA you already have. Then your phoneNumberId routing works perfectly:

if (phoneNumberId === HEALTHCARE_PHONE_NUMBER_ID) {
  return processHealthcareMessage(...)   // real patients → untouched
}
if (phoneNumberId === PHARMACY_TEST_NUMBER_ID) {
  return processPharmacyMessage(...)     // your test → isolated
}
// else → BTBIZ leadgen

Healthcare bot stays live for real patients. You test pharmacy freely. Zero interference. This is the right architecture — and it's already built in your codebase.

Bottom line: The phoneNumberId approach you thought of is exactly correct. You just need a different phone number for it to work. Don't share the Healthcare number — the cost of breaking live patient bookings is too high even for a few hours of testing.
